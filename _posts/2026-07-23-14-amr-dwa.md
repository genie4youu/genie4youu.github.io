Dynamic Window Approach(DWA)는 로봇이 지금 당장 낼 수 있는 속도 후보를 만들고, 각 후보를 짧은 시간 앞으로 시뮬레이션해 안전성과 진행도를 비교한다.

## dynamic window

로봇의 전체 속도 한계가

$$
v\in[v_{\min},v_{\max}],\qquad
\omega\in[\omega_{\min},\omega_{\max}]
$$

이고 현재 명령이 $(v_k,\omega_k)$라 하자. 한 step에서 가속도 한계로 도달할 수 있는 범위는 다음과 같다.

$$
v\in[v_k-a_vT_s,\;v_k+a_vT_s]
$$

$$
\omega\in[\omega_k-a_\omega T_s,\;\omega_k+a_\omega T_s]
$$

두 범위의 교집합이 이번 step의 dynamic window다. 따라서 갑자기 최대속도나 반대 회전으로 뛰는 후보는 처음부터 생성되지 않는다.

## 후보 trajectory rollout

각 $(v,\omega)$ 후보를 prediction horizon 동안 적분한다.

```text
dynamic window 계산
→ v, omega sampling
→ trajectory rollout
→ footprint collision 검사
→ braking trajectory 검사
→ cost 계산
→ 최저 비용 command
```

endpoint 하나만 보는 것이 아니라 모든 rollout pose에서 robot footprint가 local costmap과 충돌하는지 확인한다.

## 멈출 수 있는 후보만 admissible하다

prediction horizon 안에서는 충돌하지 않아도 현재 속도로는 장애물 전에 멈출 수 없을 수 있다. 프로젝트는 rollout collision과 별도로 braking trajectory collision을 hard rejection했다.

단순한 직선 제동거리 직관은 다음과 같다.

$$
d_{stop}\approx\frac{v^2}{2a_{brake}}
$$

실제 회전 command에서는 예상 원호를 따라 감속 trajectory를 확인해야 한다.

## cost를 분리해 튜닝한다

프로젝트의 총 cost는 다음 항목을 조합한다.

- global path와의 거리
- goal까지의 진행
- heading 정렬
- 속도 보상
- 이전 명령과의 smoothness

| 항목 | weight |
| --- | ---: |
| path | `3.0` |
| goal | `0.35` |
| heading | `0.85` |
| speed reward | `0.55` |
| smoothness | `0.20` |

각 항목은 scale이 다르므로 정규화 없이 weight 숫자만 비교하면 의미가 없다. collision과 제동 불가처럼 절대 허용할 수 없는 조건은 큰 soft cost가 아니라 hard rejection으로 분리한다.

## 프로젝트 설정

| 항목 | 값 |
| --- | ---: |
| prediction horizon | `1.20 s` |
| rollout step | `0.10 s` |
| linear samples | `5` |
| angular samples | `11` |
| 선속도 | `0 ~ 0.65 m/s` |
| 각속도 | `±1.20 rad/s` |
| 선가속도 | `1.20 m/s²` |
| 각가속도 | `4.00 rad/s²` |

standalone 시험에서 80 step 동안 4,400개의 valid candidate를 평가해 비충돌 진행을 확인했다.

## 통합 장애물 결과

사무실, 병원, 창고의 돌발 장애물 scenario에서 각각 `34.40 s`, `40.00 s`, `56.50 s`에 배송을 끝냈다. integrated obstacle run에서는 DWA mode가 227 sample 활성화되었다.

연속 command의 최대 변화는 선속도 `0.060 m/s`, 각속도 `0.200 rad/s`였고 설정한 가속도 step 안에 있었다. held/delayed LiDAR에서도 DWA와 독립 safety gate를 함께 검사했다.

## 아직 남은 실패 mode

- 모든 후보가 차단된 no-valid-candidate
- 좌우 회전을 반복하는 oscillation
- U자 장애물의 local minimum
- local obstacle가 사라진 뒤 decay
- retry 횟수와 recovery escalation

DWA가 안전한 후보를 찾지 못하면 억지로 덜 나쁜 명령을 골라서는 안 된다. zero command와 명시적 failure status를 Stateflow에 보내야 한다.
