[지난 글](/posts/01-why-state-machine/)에서 `if` 문의 한계를 봤다. 이제 실제로 Chart를 만든다.

배터리 요구사항은 이렇다.

- 외부 전원이 연결되면 충전하고, 아니면 방전한다
- 충전률 4%/스텝, 방전률 3%/스텝
- 충전 중 출력 0W, 방전 중 출력 3.5W

## State 두 개와 Transition 두 개

```mermaid
stateDiagram-v2
    direction LR
    [*] --> Charge
    Charge --> Discharge : [not isCharging]
    Discharge --> Charge : [isCharging]
```

State가 두 개, 그 사이를 오가는 Transition이 두 개다.

`[*]` 에서 `Charge` 로 들어가는 화살표는 **Default Transition** 이다. 편집기에서는 파란 원으로 보인다. 시뮬레이션이 시작될 때 어느 State가 먼저 active 되는지를 지정하며, 첫 번째로 추가한 State에 자동으로 붙는다. 이게 없으면 Chart는 시작할 State를 모른다.

State 이름에는 규칙이 있다. 공백을 쓸 수 없고, 숫자로 시작할 수 없으며, 각 이름이 고유해야 한다. State 경계가 서로 겹쳐서도 안 된다.

## Transition 라벨은 세 부분으로 나뉜다

Transition에 붙는 라벨에는 정해진 형식이 있다.

```text
Event [Condition] {Action}
```

| 부분 | 표기 | 뜻 |
| --- | --- | --- |
| **Event** | 그대로 씀 | 이 Event가 오기 전에는 평가하지 않는다 |
| **Condition** | `[ ... ]` | 이 조건이 참이어야 Transition이 일어난다 |
| **Action** | `{ ... }` | Transition이 일어날 때 실행되는 동작 |

셋 다 선택이다. 배터리 예제에서는 Condition만 쓴다.

Event와 Condition은 역할이 다르다. Event는 "언제 평가할 것인가"를 정하고, Condition은 "넘어가도 되는가"를 정한다. 지금은 크게 와닿지 않지만 [6편](/posts/06-parallel-and-events/)에서 이 차이가 작동한다.

## State Action: entry, during, exit

State 안에도 코드를 넣는다. 언제 실행되는지가 키워드로 갈린다.

| 키워드 | 실행 시점 |
| --- | --- |
| `entry` | State가 active 되는 순간 (한 번) |
| `during` | State가 active인 매 스텝 |
| `exit` | State가 inactive 되는 순간 (한 번) |

배터리에 적용하면 이렇게 된다.

```text
Charge
  entry:  sentPower = 0;
  during: charge = charge + 4;

Discharge
  entry:  sentPower = 3.5;
  during: charge = charge - 3;
```

왜 `sentPower` 는 `entry` 이고 `charge` 는 `during` 인가. 출력은 모드가 바뀌는 순간 한 번만 정하면 되지만, 충전량은 시간이 흐르는 동안 계속 변하기 때문이다. 한 번만 할 일과 계속 할 일을 문법으로 구분하는 셈이다. `if` 문에서는 이 구분을 직접 관리해야 했다.

```mermaid
flowchart LR
    A[Discharge active] --> B[exit]
    B --> C[Transition]
    C --> D["entry: sentPower = 0"]
    D --> E[Charge active]
    E --> F["during: charge += 4"]
    F --> E
```

> `during` 은 State에 머무는 동안 항상 도는 코드가 아니다. active나 inactive가 되는 스텝에는 실행되지 않고, 유효한 Transition이 있으면 실행조차 되지 않는다. 이건 [2부에서 따로](/posts/stateflow-during-and-chart-lifecycle/) 다룬다.

## Chart Data: Input, Output, Local

Transition이나 Action에서 변수를 쓰려면 반드시 Data로 정의해야 한다. 정의하지 않으면 경고 배지가 뜬다.

| 종류 | 하는 일 | Simulink에 미치는 영향 |
| --- | --- | --- |
| **Input** | 바깥에서 값을 받는다. 차트가 수정할 수 없다 | Chart 블록에 입력 포트가 생긴다 |
| **Output** | 차트가 바깥으로 값을 내보낸다 | Chart 블록에 출력 포트가 생긴다 |
| **Local** | 차트 내부에서만 쓰는 저장값 | 포트가 없다 |

배터리 예제의 세 변수를 분류하면 이렇다.

| 변수 | 종류 | 이유 |
| --- | --- | --- |
| `isCharging` | Input | 외부 전원 연결 여부. 차트는 읽기만 한다 |
| `sentPower` | Output | 차트가 계산해 바깥으로 내보낸다 |
| `charge` | Local | 차트 내부의 충전량. 바깥에서 볼 필요가 없다 |

```mermaid
flowchart LR
    IN[isCharging - Input] --> CH[Chart: Battery]
    CH --> OUT[sentPower - Output]
    LOC[charge - Local] -.내부에만 존재.-> CH
```

Stateflow는 문맥으로 타입을 추론한다. Symbols 창의 Resolve undefined symbols를 누르면 위 분류를 자동으로 제안해준다. 다만 초기값은 직접 넣어야 한다. `charge` 를 50으로 두지 않으면 시뮬레이션이 0에서 시작한다.

## Simulink에 연결하기

Chart는 혼자 돌지 않는다. Simulink 모델 안의 Chart 블록으로 들어간다.

입력으로 `Constant(1)` 과 `Constant(0)` 을 Manual Switch에 물려 `isCharging` 에 연결하면, 시뮬레이션 중에 손으로 전원을 껐다 켰다 할 수 있다. 출력 `sentPower` 는 Scope에 연결한다. Stop Time을 `Inf` 로 두면 계속 돌면서 스위치를 만질 수 있다.

Run을 누르면 active State의 테두리가 파랗게 강조된다. 스위치를 토글하면 강조가 옮겨간다. 로직이 눈에 보인다. 이게 Stateflow를 쓰면서 처음 느끼는 이점이다.

## 정리

| 개념 | 핵심 |
| --- | --- |
| **Default Transition** | 시작할 State를 지정하는 파란 원 |
| **Transition 라벨** | `Event [Condition] {Action}`, 셋 다 선택 |
| **State Action** | `entry`(진입), `during`(매 스텝), `exit`(이탈) |
| **Chart Data** | Input(읽기만), Output(내보냄), Local(내부) |
