[지난 글](/posts/03-log-and-debug/)에서 결함을 찾았다. 충전량이 **100%를 넘고, 0% 아래로 내려간다.**

고치는 방법은 두 갈래다.

| 방법 | 어떻게 |
| --- | --- |
| ❌ 조건문 추가 | `during: if (charge < 100) charge += 4;` |
| ✅ **구조 변경** | `Charge` State **안에** 세부 모드를 만든다 |

첫 번째가 더 빨라 보인다. 하지만 그건 [1편에서 버린 방식](/posts/01-why-state-machine/) — 모드를 다시 변수 조합 속에 숨기는 것이다. **"완충 상태"라는 모드가 코드에 드러나지 않는다.**

Stateflow 답게 풀어보자.

---

## 1. State 안에 State를 넣는다

State는 다른 State를 **품을 수 있다.** 바깥을 **Parent**, 안을 **Child**(또는 substate)라고 부른다.

```mermaid
stateDiagram-v2
    direction LR
    [*] --> Charge

    state Charge {
        direction LR
        [*] --> FastCharge
        FastCharge --> SlowCharge : [charge > 80]
        SlowCharge --> Full : [charge == 100]
    }

    state Discharge {
        direction LR
        [*] --> Powered
        Powered --> Empty : [charge <= 3]
    }

    Charge --> Discharge : [~isCharging]
    Discharge --> Charge : [isCharging]
```

바깥 구조(`Charge` ↔ `Discharge`)는 **그대로**다. 각 State 안에 세부 모드가 생겼을 뿐이다.

### 계층의 규칙

| 규칙 | 뜻 |
| --- | --- |
| Parent가 active 되면 | 내부 **Default Transition** 이 가리키는 Child가 active 된다 |
| Parent가 inactive 되면 | **Child도 전부 inactive** 된다 |
| 배치 | Child는 Parent 안에 **완전히 포함**돼야 하고, 경계가 겹치면 안 된다 |

---

## 2. 충전을 세 모드로 쪼갠다

| Parent | Child | `during` | 나가는 조건 |
| --- | --- | --- | --- |
| `Charge` | **`FastCharge`** | `charge = charge + 4` | `[charge > 80]` → `SlowCharge` |
| `Charge` | **`SlowCharge`** | `charge = charge + 1` | `[charge == 100]` → `Full` |
| `Charge` | **`Full`** | *(없음)* | *(없음)* |

`Full` State에는 **`during` Action이 없다.** 그래서 `charge` 가 더 이상 증가하지 않는다.

> **조건문으로 "더하지 마라"를 막은 게 아니라, 더하는 코드가 없는 State로 옮겨간 것이다.**
>
> 차이가 미묘해 보이지만 결과는 크다 — 이제 **"완충"이 이름 붙은 모드로 존재한다.**
> 로그를 보면 `Battery.Charge.Full` 이 찍힌다. 추론할 필요가 없다.
{: .prompt-tip }

방전 쪽도 같다.

| Parent | Child | `during` | 나가는 조건 |
| --- | --- | --- | --- |
| `Discharge` | **`Powered`** | `charge = charge - 3` | `[charge <= 3]` → `Empty` |
| `Discharge` | **`Empty`** | *(없음)* · `entry: sentPower = 0` | *(없음)* |

`Empty` 는 `entry` 에서 **출력을 끊는다.** 배터리가 바닥났으니 전력을 못 준다. 이것도 조건문이 아니라 **State의 `entry` Action** 으로 표현된다.

---

## 3. 계층이 실제로 절약하는 것

계층을 쓰면 뭐가 좋은가? "정리가 된다"는 막연한 말 말고, 구체적으로.

### ① 바깥 Transition을 한 번만 그린다

`Charge` 안에 세 개의 Child가 있다. 전원이 빠지면(`~isCharging`) 셋 다 `Discharge` 로 가야 한다.

계층이 **없다면** — `FastCharge → Discharge`, `SlowCharge → Discharge`, `Full → Discharge` 로 **화살표 세 개**를 그려야 한다.

계층이 **있다면** — `Charge → Discharge` **하나면 된다.** Parent에서 나가는 Transition은 **어느 Child에 있든** 적용된다.

```mermaid
flowchart LR
    subgraph N["❌ 계층 없이"]
        direction TB
        F1["FastCharge"] --> D1["Discharge"]
        S1["SlowCharge"] --> D1
        U1["Full"] --> D1
    end
    subgraph Y["✅ 계층으로"]
        direction TB
        subgraph C2["Charge"]
            F2["FastCharge"]
            S2["SlowCharge"]
            U2["Full"]
        end
        C2 --> D2["Discharge"]
    end

    style N stroke:#e74c3c,stroke-width:2px
    style Y stroke:#27ae60,stroke-width:2px
```

Child가 10개로 늘어도 바깥 화살표는 **여전히 하나**다. 이게 계층이 막아주는 **조합 폭발**이다.

### ② 공통 동작을 Parent에 모은다

"충전 중에는 출력이 0" — 이건 세 Child 모두에 해당한다. 그래서 `Charge` **Parent의 `entry`** 에 `sentPower = 0` 을 둔다. Child마다 반복하지 않는다.

### ③ 관심사가 분리된다

- **바깥 계층**: "충전 중인가, 방전 중인가" — 큰 그림
- **안쪽 계층**: "충전 중에 얼마나 빠르게 넣는가" — 세부

읽는 사람이 필요한 층만 보면 된다. Chart를 접었다 폈다 할 수 있는 이유다.

## 4. 고쳤다. 그런데 또 문제가 있다

이제 `charge` 는 0~100을 벗어나지 않는다. 요구사항 하나를 지켰다.

그런데 방전 쪽을 다시 보자.

```text
Powered
  entry:  sentPower = 3.5;
  during: charge = charge - 3;
```

**기기가 요구하는 전력이 3.5W보다 작으면?** 예를 들어 1W만 필요한데도, 배터리는 **항상 3.5W를 내보내고 항상 3%씩 깎인다.**

출력이 **수요에 반응하지 않는다.** 상수로 박혀 있다.

> 이건 State를 하나 더 만들어서 풀 수 있는 문제가 아니다.
> **"수요가 한계보다 크면 한계만큼, 아니면 수요만큼"** 이라는 **계산**이 필요하다.
>
> State 안에서 갈래를 나눠야 한다. 그게 **Junction** 이다.
{: .prompt-warning }

---

## 정리

| | |
| --- | --- |
| **Parent / Child** | State는 State를 품는다. Parent가 꺼지면 Child도 다 꺼진다 |
| **버그를 구조로 푼다** | `if` 를 추가하는 대신, **동작이 없는 State**(`Full`)로 옮긴다 |
| **계층이 절약하는 것** | 바깥 Transition을 **하나로** · 공통 Action을 **Parent에** · 관심사 분리 |

> **한 줄로:** 계층은 정리 정돈이 아니라 **화살표 개수를 곱셈에서 덧셈으로 바꾸는 장치**다.
{: .prompt-tip }
