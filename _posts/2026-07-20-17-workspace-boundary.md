[13편 §7](/posts/13-mcp-next-steps/)에서 "구조로 좁힌다"까지 정리하고 구체적인 설계는 미뤄뒀다. 이 글이 그 결론이다.

## 1. 문제의 형태

에이전트에게 디렉터리를 열어주면 **그 안의 파일 내용이 모델 제공자의 서버로 간다.**

로컬 MCP 서버를 쓴다고 달라지지 않는다. MCP 서버는 도구를 제공할 뿐이고, **대화 자체는 클라우드 모델이 처리**한다([02편](/posts/02-mcp-architecture/)). 서버가 로컬이라는 사실과 대화가 외부로 나간다는 사실은 별개다.

## 2. 방어가 아닌 것 세 가지

### Roots

[04편](/posts/04-mcp-primitives/)의 Roots는 클라이언트가 서버에게 작업 범위를 알려주는 장치다. 여기에 기대고 싶어지지만 **스펙이 직접 아니라고 말한다.**

> "Roots serve as a **coordination mechanism** between clients and servers, **not a security boundary**."
> "Actual security must be enforced at the **operating system level**, via file permissions and/or sandboxing."

### 규칙 파일

에이전트가 읽는 규칙 파일에 "이 폴더 밖은 보지 마라"라고 써도, 그것은 **협조 요청이지 강제가 아니다.** 모델이 지킬 수도 있고 안 지킬 수도 있다. Roots가 방어가 아닌 것과 정확히 같은 이유다.

### 설정의 신뢰 항목

[16편 §4](/posts/16-mcp-config-applied/)에서 확인한 대로 `trust_level`은 **승인 절차**를 정하지 접근 권한을 정하지 않는다. 항목을 지워도 열면 읽는다.

```
❌ 규칙 파일에 범위를 적는다      → 협조 요청
❌ 설정에서 신뢰 항목을 지운다     → 승인 절차만 바뀐다
✅ 그 디렉터리를 열지 않는다       → 실제 경계
```

## 3. 실제 경계는 하나 — 워크스페이스

에이전트가 읽을 수 있는 범위를 정하는 것은 **어떤 디렉터리를 워크스페이스로 여느냐** 하나다.

그리고 [14편 §3](/posts/14-mcp-editor-integration/)에서 확인한 것이 여기에 더해진다. **에디터 확장은 열려 있는 파일을 프롬프트에 함께 넣는다.** 그래서 경계는 두 개 항목으로 결정된다.

```
노출 범위 = (워크스페이스로 연 디렉터리) + (그 순간 편집기에 떠 있던 것)
```

에디터 확장은 **창 단위로 워크스페이스를 잡는다.** 그래서 실무에서는 **창을 나누는 것**이 경계가 된다.

## 4. 채택한 구조

### (1) 실습은 별도 디렉터리에서

에이전트로 하는 작업은 **처음부터 분리된 디렉터리**에서 한다. 노트·프롬프트 기록·실험 기록도 그 안에 둔다.

```
practice/
├─ AGENTS.md          규칙
├─ notes/             개요 · prompts/ · experiments/
├─ exchange/          교환소
└─ (작업 대상 파일)
```

규칙 파일을 두는 이유는 그것이 방어라서가 아니다. **방어는 디렉터리 분리가 하고, 규칙 파일은 그 안에서의 작업 품질을 위한 것**이다. 둘을 구분해야 규칙 파일에 과도하게 기대지 않는다.

### (2) 교환은 손으로

두 공간 사이에 파일을 옮길 일이 생긴다. 이때 **자동 동기화를 걸지 않는다.** 심볼릭 링크, 공유 폴더, 동기화 스크립트 전부 만들지 않는다.

```
문서 쪽/_교환/to-practice/    ──── 손으로 복사 ────▶  practice/exchange/from-vault/
문서 쪽/_교환/from-practice/  ◀─── 손으로 복사 ────   practice/exchange/to-vault/
```

**두 디렉터리는 연결되어 있지 않다.** 교환소가 하는 일은 "지금 옮기려는 것이 무엇인지 한곳에 모아서, 옮기기 전에 한 번 보게" 만드는 것뿐이다.

> **자동화하면 사람이 한 번 보는 검문소가 사라진다.** 그 검문소가 실제 방어선이다.
> 폴더 이름은 `보낼것/받은것` 대신 **방향(`to-`/`from-`)** 으로 짓는다. 양쪽에서 의미가 뒤집히지 않는다.

### (3) 목록으로 지정하고, 기본값은 자유

처음에는 "이 폴더들은 통째로 금지"로 시작했다. **이 방식은 오래 못 간다.**

과잉 차단이 되면 규칙이 일상 작업을 방해하고, 그러면 **규칙 자체를 무시하게 된다.** 그 시점에 진짜 위험한 것도 함께 통과한다.

그래서 **민감한 것만 목록으로 지정하고 나머지는 자유**로 뒤집었다. 목록 파일 하나를 두고 규칙 파일이 그것을 가리키게 한다.

**목록은 사람이 관리한다.** 에이전트가 짐작으로 넓히지 않는다. 애매한 것을 발견하면 목록에 추가하는 대신 **물어본다.**

> 이것은 [16편 §3](/posts/16-mcp-config-applied/)의 승인 피로와 같은 구조다.
> 경고가 잦으면 사람이 읽지 않고, 읽지 않으면 방어선이 죽는다.
> **미탐만 위험한 것이 아니라 과잉 차단도 위험하다.**

## 5. 남은 한계

정직하게 적어둔다.

- **위 방어는 전부 사람이 지키는 규칙이다.** 스펙이 말한 OS 수준 강제(파일 권한·샌드박스)가 아니다.
- 창을 잘못 열면 그대로 뚫린다. **실수 한 번을 막아주는 장치가 없다.**
- 더 단단하게 하려면 별도 사용자 계정이나 파일 권한으로 막아야 한다. **편의와의 거래이고, 아직 그 거래를 하지 않았다.**

이 한계를 적어두는 이유는, 구조를 만들고 나면 **실제보다 안전하다고 느끼기 때문이다.** 폴더를 나누고 목록을 만들면 일이 끝난 것처럼 보이지만, 강제력의 층위는 그대로다.

## 📌 정리

- **Roots도 규칙 파일도 설정의 신뢰 항목도 방어가 아니다.** 스펙이 직접 그렇게 말한다
- 실제 경계는 **워크스페이스로 무엇을 여느냐** — 여기에 **편집기에 무엇이 떠 있었느냐**가 더해진다
- 작업은 **별도 디렉터리**에서. 규칙 파일은 방어가 아니라 작업 품질을 위한 것
- 교환은 **손으로**. 자동 동기화는 사람이 보는 검문소를 없앤다
- **기본값은 자유, 민감한 것만 목록으로.** 과잉 차단은 규칙 자체를 무시하게 만든다
- 이 전부가 **규칙 기반이다.** OS 수준 강제가 아니라는 한계를 알고 쓴다
